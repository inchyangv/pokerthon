# PROJECT.md

## 1. 문서 목적

이 문서는 **외주 개발자 또는 Claude Code**가 그대로 구현 작업에 들어갈 수 있도록 작성한 최종 기획/개발 기준 문서다.

이 문서에 적힌 내용이 우선이다. 포커 일반 규칙과 다른 부분이 있더라도, **이 프로젝트의 커스텀 룰이 우선**한다.

---

## 2. 프로젝트 한 줄 정의

**여러 AI 에이전트가 외부 API로 접속해서 플레이할 수 있는 멀티테이블 텍사스 홀덤 플랫폼을 만든다.**

핵심 조건은 다음과 같다.

- 외부 접속 가능 API 제공
- 1개 테이블 최대 9명
- 전체 참가자는 대략 15명 수준
- 멀티테이블 동시 운영 지원
- 자기 홀카드 + 공개 정보 조회 가능
- 자기 턴에 액션 제출 가능
- 올인, 사이드팟 필수
- 스트래들 없음
- 관리자 지급 칩 방식
- 복잡한 인증 대신 `API_KEY / SECRET_KEY` 서명 방식 사용
- 테이블 번호를 지정해서 앉기 / 일어서기 가능
- 테이블별 게임 이력, 액션 이력 조회 가능
- 액션 10분 이상 없으면 자동 폴드
- Railway 배포 기준

## 3. 이번 버전의 성격

이번 버전은 **정식 MTT(멀티테이블 토너먼트 엔진)** 가 아니라,

**"관리자 지급 칩 + 여러 테이블에서 실시간으로 플레이 가능한 AI 홀덤 플랫폼"** 으로 정의한다.

즉, 이번 범위에 포함되는 것은:

- 여러 테이블 동시 운영
- 계정/키 발급
- 관리자 칩 지급
- 지정 테이블 착석/이석
- 핸드 진행
- 액션/게임 로그 저장 및 조회
- 외부 봇 API

이번 범위에서 제외되는 것은:

- 자동 블라인드 레벨 상승
- 자동 테이블 밸런싱
- 파이널 테이블 자동 편성
- 정식 토너먼트 브래킷/랭킹 엔진
- 실시간 채팅
- 스트래들
- 복잡한 사용자 인증 체계(OAuth, JWT, 소셜 로그인 등)

필요하면 차후 2차 개발로 분리한다.

## 4. 반드시 고정할 정책

### 4.1 플레이 기본 규칙

- 게임 종류: **No-Limit Texas Hold'em**
- 테이블당 최대 좌석 수: **9석**
- 최소 시작 인원: **2명**
- 스트래들: **미지원**
- 블라인드: **1 / 2**
- 안테: **없음**
- 기본 테이블 바이인: **40칩 고정**
- 계정 생성 시 기본 칩 지급: **0칩**
- 칩은 **관리자가 계정에 지급**한다.

### 4.2 계정 칩과 테이블 칩의 관계

칩은 두 층으로 관리한다.

1. **계정 지갑(wallet balance)**
   - 계정이 보유한 총 칩
   - 계정 생성 시 0
   - 관리자만 증감 가능

2. **테이블 스택(table stack)**
   - 특정 테이블에 앉았을 때 실제 게임에 쓰는 칩
   - 착석 시 계정 지갑에서 **40칩 차감** 후 테이블 스택 40으로 시작
   - 일어서면 남은 테이블 스택을 계정 지갑으로 반환

즉, 계정에 최소 40칩 이상 있어야 착석 가능하다.

### 4.3 착석 / 이석 정책

- 사용자는 **테이블 번호를 지정**해서 앉을 수 있어야 한다.
- 좌석 번호까지 지정 가능하면 더 좋다. 좌석 번호를 안 주면 서버가 빈 자리 중 하나를 자동 배정해도 된다.
- **좌석 번호 범위: 1~9** (0이 아닌 1부터 시작)
- 손(핸드) 진행 중에는 즉시 이석하지 않고, **현재 핸드 종료 후 이석** 처리한다.
- 핸드가 진행 중이지 않으면 즉시 이석 가능하다.
- 스택이 0이 되면 핸드 종료 후 자동 이석 처리한다.
- **핸드 진행 중 착석한 플레이어는 현재 핸드에 참여하지 않고, 다음 핸드부터 참여한다.**
- 같은 테이블에 **이미 앉아 있는 상태에서 다시 착석 요청하면 에러** 반환한다.

### 4.4 액션 타임아웃

- 플레이어의 턴이 시작된 시점부터 **10분(600초)** 안에 액션이 없으면 서버가 자동으로 `FOLD` 처리한다.
- 이 자동 액션은 반드시 로그에 남겨야 한다.
- 로그 액션 타입은 예: `AUTO_FOLD_TIMEOUT`
- 체크 가능한 상황이라도 예외 없이 `FOLD` 처리한다. 이번 프로젝트는 단순성을 우선한다.
- 연속 타임아웃에 대한 자동 킥/차단 정책은 이번 버전에서는 **없음**. 매번 폴드 처리만 한다.
- 타임아웃 체크는 **서버 측 백그라운드 태스크(주기적 폴링 또는 스케줄러)** 로 구현한다. 정밀한 600초일 필요는 없고, ±수초 오차는 허용한다.

### 4.5 체크레이즈 / 콜레이즈 정책

- **체크레이즈 허용**
- **콜레이즈 금지**

API 관점에서 해석하면 다음과 같다.

- 한 턴에 제출 가능한 액션은 정확히 하나다.
- `CALL` 후 같은 턴에 이어서 `RAISE` 하는 복합 액션은 허용하지 않는다.
- 레이즈는 반드시 `RAISE_TO(total_amount)` 형태로 한 번에 제출해야 한다.
- 체크 후 액션이 다시 돌아왔을 때 레이즈하는 상황(체크레이즈)은 허용한다.

### 4.6 최소 레이즈 규칙 (커스텀 룰)

이 프로젝트는 일반적인 표준 NLHE 최소 레이즈 규칙 대신, 아래 커스텀 룰을 사용한다.

- 현재 스트리트에 **아직 베팅이 없는 경우**, 최소 베팅은 **2칩**(빅블라인드)
- 이미 베팅이 존재하는 경우, 최소 레이즈 총액은:

```text
min_raise_to = ceil(current_highest_bet * 1.5)
```

예시:

- 현재 최고 베팅이 2이면 최소 레이즈 총액은 3
- 현재 최고 베팅이 10이면 최소 레이즈 총액은 15
- 현재 최고 베팅이 17이면 최소 레이즈 총액은 26

**레이즈 횟수 제한: 없음.** No-Limit이므로 한 스트리트 내 레이즈 횟수에 상한을 두지 않는다.

### 4.7 올인 예외

- 올인은 언제나 허용한다.
- 플레이어의 남은 스택이 최소 레이즈 기준보다 부족하면, **최소 레이즈 미만 올인**도 허용한다.
- 단, 이런 숏 올인은 **레이즈 권한을 다시 열지 않는다**.
- 여러 숏 올인이 누적되더라도 이번 버전에서는 **raise reopen 계산을 단순화**해서, 최소 레이즈를 충족하지 못한 올인은 기존 액터의 추가 레이즈 권한을 열지 않는 것으로 처리한다.
- **CALL 시 스택이 부족한 경우**: `to_call` 금액보다 남은 스택이 적으면 자동으로 **올인(콜 올인)** 처리한다. 이 경우 사이드팟이 생성된다.
- **모든 칩 연산은 정수(integer)** 로 처리한다. 소수점은 발생하지 않는다.

### 4.8 사이드팟

- **사이드팟 필수 구현**
- 팟은 실시간 소스 오브 트루스로 직접 쪼개서 관리하기보다,
  **플레이어별 hand contribution(해당 핸드 총 투입 칩)** 을 기준으로 계산하는 방식이 좋다.
- 메인팟 / 사이드팟 / 언콜 금액 반환을 정확히 처리해야 한다.

### 4.9 카드 표기법

카드는 **2글자 문자열**로 표기한다.

- 첫 글자: 랭크 — `2`, `3`, `4`, `5`, `6`, `7`, `8`, `9`, `T`, `J`, `Q`, `K`, `A`
- 둘째 글자: 수트 — `s`(스페이드), `h`(하트), `d`(다이아), `c`(클럽)

예시: `As`(에이스 스페이드), `Td`(텐 다이아), `2c`(투 클럽)

### 4.10 덱 셔플

- **표준 Fisher-Yates 셔플** 사용
- Python의 경우 `random.shuffle()` 사용 가능 (Mersenne Twister 기반, 이 프로젝트에서는 충분)
- 암호학적 보안 수준의 RNG는 요구하지 않는다

### 4.11 핸드 랭킹

표준 텍사스 홀덤 핸드 랭킹을 사용한다 (높은 순서):

1. Royal Flush
2. Straight Flush
3. Four of a Kind
4. Full House
5. Flush
6. Straight
7. Three of a Kind
8. Two Pair
9. One Pair
10. High Card

- `A`는 스트레이트에서 하이(A-K-Q-J-T)와 로우(5-4-3-2-A) 모두 사용 가능
- 5-4-3-2-A 스트레이트(wheel)는 가장 낮은 스트레이트
- 수트 간 우열은 없다

## 5. 사용자 역할

### 5.1 관리자(Admin)

관리자가 할 수 있어야 하는 것:

- 참가자 계정 생성
- 닉네임 지정
- API_KEY / SECRET_KEY 발급

- 칩 지급 / 차감
- 테이블 생성
- 테이블 상태 조회
- 테이블별 이력 조회

### 5.2 플레이어(Player / Bot)

플레이어는 외부 API를 통해 다음을 할 수 있어야 한다.

- 자기 계정 정보 조회
- 자기 칩 잔액 조회
- 특정 테이블 상태 조회
- 특정 테이블에 착석
- 특정 테이블에서 이석

- 자기 핸드 + 공개 정보 조회

공개 조회로 다음 정도는 볼 수 있어야 한다.

- 테이블 목록

- 보드 카드
- 팟/사이드팟 현황

## 6. 핵심 기능 요구사항

## 6.1 계정 생성 및 키 발급

### 요구사항

- 관리자는 관리자 페이지에서 참가자 계정을 생성할 수 있어야 한다.
- 계정 생성 시 **닉네임**을 입력해야 한다.
- 닉네임은 중복 불가가 좋다.
- 계정 생성 시 기본 칩은 0이다.
- 계정에 대해 **API_KEY / SECRET_KEY** 를 생성할 수 있어야 한다.
- SECRET_KEY는 생성 직후 1회만 보여주고, 이후에는 다시 보여주지 않아도 된다.
- 키는 폐기 및 재발급 가능해야 한다.

### 권장 정책

- 계정당 활성 키쌍은 1개만 허용해도 충분하다.
- 재발급 시 기존 키는 즉시 비활성화한다.

## 6.2 인증 방식

복잡한 로그인/세션/JWT 대신, 모든 플레이어 API는 다음 헤더 기반 인증을 사용한다.

### 요청 헤더

- `X-API-KEY`
- `X-TIMESTAMP`
- `X-NONCE`
- `X-SIGNATURE`

### 서명 방식

서명 알고리즘은 **HMAC-SHA256** 로 고정한다.

서명 원문(canonical string)은 아래 형식으로 고정한다.

```text
{timestamp}\n
{nonce}\n
{http_method}\n
{request_path}\n
{canonical_query_string}\n
{sha256_hex(body)}
```

실제 구현에서는 빈 줄 없이 다음 형식으로 만드는 것을 권장한다.

```text
{timestamp}\n{nonce}\n{method}\n{path}\n{query_string}\n{body_sha256}
```

예시:

```text
1710000000
abc-123
POST
/v1/private/tables/1/action

3f79bb7b435b05321651daefd374cd21b4...
```

그리고 다음처럼 계산한다.

```text
signature = hex(HMAC_SHA256(secret_key, canonical_string))
```

### 세부 사양 확정

- **타임스탬프 형식**: Unix epoch 초 단위 (정수, 10자리). 밀리초가 아님.
- **nonce 형식**: 임의 문자열. UUID v4를 권장하지만 강제하지 않는다. 최대 길이 64자.
- **canonical_query_string 생성 규칙**:
  - 쿼리 파라미터를 키 기준 알파벳 오름차순 정렬
  - `key1=value1&key2=value2` 형태로 연결
  - 쿼리 파라미터가 없으면 빈 문자열 `""`
  - URL 인코딩된 값은 그대로 사용 (디코드하지 않음)
- **body SHA256**: 요청 바디의 SHA-256 해시 (hex, lowercase). **바디가 없는 경우(GET 등)** 빈 문자열(`""`)의 SHA-256을 사용한다: `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`

### 검증 규칙

- 타임스탬프 오차 허용 범위: **±300초 (5분)** — 확정
- `nonce` 재사용 금지
- 만료/폐기된 키는 거절
- 서명 불일치 시 401
- 타임스탬프 범위 이탈 시 401
- 동일 nonce 재사용 시 401 또는 409
- **nonce 보관 정책**: 타임스탬프 + 5분 이후의 nonce는 더 이상 충돌 검사할 필요 없으므로, **10분 이상 지난 nonce는 주기적으로 삭제** 가능 (DB 비대화 방지)

### 비고

- 이 프로젝트는 내부 대회용이므로 OAuth/JWT는 구현하지 않는다.
- 관리자 페이지는 별도 단순 비밀번호 로그인으로 보호해도 된다.
- **관리자 API(`/admin/*`)는 HMAC 서명 대신 세션 쿠키 또는 `Authorization: Bearer {ADMIN_PASSWORD}` 헤더 방식**으로 인증한다.

## 6.3 칩 지급/차감

### 요구사항

- 관리자는 특정 계정에 칩을 지급할 수 있어야 한다.
- 관리자는 특정 계정에서 칩을 차감할 수 있어야 한다.
- 모든 칩 변동은 **ledger(원장)** 로 기록해야 한다.
- 칩 변동 사유를 텍스트로 남길 수 있어야 한다.

### 예시 이벤트

- `ADMIN_GRANT` — 관리자 칩 지급
- `ADMIN_DEDUCT` — 관리자 칩 차감
- `TABLE_BUYIN` — 테이블 착석 시 지갑→테이블 차감
- `TABLE_CASHOUT` — 테이블 이석 시 테이블→지갑 반환
- `HAND_WIN` — 핸드 승리로 인한 칩 증가
- `HAND_LOSS` — 핸드에서 잃은 칩 (기록용, 실제 차감은 베팅 시점에 이미 발생)
- `MANUAL_ADJUST` — 수동 조정

### 칩 차감 정책

- **관리자 차감 시 지갑 잔액이 차감액보다 적으면 에러** 반환한다. 지갑 잔액은 음수가 될 수 없다.
- **테이블에 앉아 있는 플레이어의 지갑에서 관리자가 차감하는 것은 허용**한다. 단, 차감 대상은 지갑 잔액이며, 테이블 스택은 건드리지 않는다.
- 핸드 승리/패배 시 칩 변동은 **테이블 스택 레벨에서만** 발생하며, 이석 시 `TABLE_CASHOUT`으로 일괄 정산된다.

## 6.4 멀티테이블 운영

### 요구사항

- 테이블을 여러 개 운영할 수 있어야 한다.
- 테이블 번호(`table_no`)는 사람이 보기 쉬운 정수 형태가 좋다. 예: 1, 2, 3 ...
- 각 테이블은 독립적으로 핸드가 진행되어야 한다.
- 한 테이블 최대 좌석은 9석이다.
- 전체 15명 정도가 동시에 참여 가능해야 하므로 최소 2개 이상 테이블 동시 운영이 가능해야 한다.

### 권장 정책

- `table_no` 는 unique
- 테이블 상태: `OPEN`, `PAUSED`, `CLOSED`
- 2명 이상 준비 상태면 자동으로 다음 핸드 시작

### 테이블 상태 전이 규칙

- `OPEN` → `PAUSED`: 관리자가 일시정지. **현재 핸드가 진행 중이면 핸드 종료 후 일시정지** 상태로 전환. 새 핸드는 시작하지 않는다.
- `PAUSED` → `OPEN`: 관리자가 재개. 2명 이상 앉아 있으면 자동으로 다음 핸드 시작.
- `OPEN` 또는 `PAUSED` → `CLOSED`: 관리자가 종료. **현재 핸드가 진행 중이면 핸드 종료 후 종료** 처리. 모든 플레이어의 테이블 스택을 지갑으로 반환하고 자동 이석 처리.
- `CLOSED` → 재오픈 불가. 새 테이블을 생성해야 한다.
- **`PAUSED` 또는 `CLOSED` 상태의 테이블에는 착석 불가.**

---

## 6.5 착석 / 이석

### 착석

- 플레이어는 특정 `table_no` 로 착석 요청 가능해야 한다.
- `seat_no` 를 선택적으로 지정할 수 있어야 한다.
- `seat_no` 미지정 시 서버가 빈 자리 자동 할당 가능
- 착석 시 계정 지갑에서 40칩 차감, 테이블 스택 40 생성
- 계정 지갑 잔액이 40 미만이면 착석 실패
- 이미 다른 테이블에 앉아 있으면 정책을 명확히 정해야 한다.

### 이번 버전 정책

- **한 계정은 동시에 하나의 테이블에만 착석 가능**

### 이석

- 현재 핸드 미참여 중이면 즉시 이석
- 현재 핸드 참여 중이면 `leave_after_hand = true` 로 표시 후 핸드 종료 시 이석
- 이석 시 남은 테이블 스택을 계정 지갑으로 반환

---

## 6.6 핸드 자동 시작 규칙

- 테이블에 **스택이 0보다 큰 플레이어가 2명 이상** 앉아 있으면 핸드 시작 가능
- 핸드가 끝난 뒤, 2명 이상 유지되면 자동으로 다음 핸드 시작
- 다음 핸드 시작까지 짧은 대기(예: 2~3초)는 있어도 좋다
- **최초 버튼 결정**: 테이블에서 첫 핸드가 시작될 때, 버튼 좌석은 **가장 낮은 seat_no를 가진 활성 좌석**에 배정한다. (랜덤 결정보다 결정론적 방식이 테스트와 복구에 유리)
- **핸드 사이에 새로 앉은 플레이어**: 다음 핸드부터 참여 가능. 블라인드 포스팅 면제 없이 정상 참여한다.
- **핸드 사이에 이석 처리된 플레이어**: `leave_after_hand` 상태였던 플레이어는 핸드 종료 시점에 스택 반환 후 제거. 남은 인원이 2명 미만이면 다음 핸드를 시작하지 않는다.

## 6.7 게임 상태 조회

플레이어는 항상 다음 정보를 조회할 수 있어야 한다.

### Private State (자기 관점)

- table_no
- hand_id
- street (`preflop`, `flop`, `turn`, `river`, `showdown`)
- 자기 홀카드
- 보드 카드
- 좌석별 공개 정보
  - seat_no
  - nickname
  - current_stack
  - folded 여부
  - all_in 여부
- dealer/button 위치
- 현재 액션 차례 seat
- 현재 최고 베팅액
- 내가 콜해야 하는 금액(`to_call`)
- 내가 가능한 액션 목록(`legal_actions`)
- 최소 레이즈 총액(`min_raise_to`)
- 최대 레이즈 총액(사실상 올인 총액)
- 메인팟 금액
- 사이드팟 목록
- 플레이어별 이번 라운드 기여금
- 플레이어별 이번 핸드 총 기여금
- 액션 데드라인 시각
- 상태 버전(`state_version`)

### Public State (공개 관점)

- 위 정보 중 자기 홀카드를 제외한 공개 정보
- 타인의 홀카드는 showdown 이전에는 절대 노출 금지

### 핸드가 없을 때의 상태

- 현재 진행 중인 핸드가 없는 경우에도 상태 조회는 가능해야 한다.
- 이 경우 `hand_id`는 `null`, `street`는 `null`, 보드/홀카드는 빈 배열, `legal_actions`는 빈 배열로 응답한다.
- 좌석/스택/버튼 위치 등 테이블 레벨 정보는 정상 반환한다.

### 중요

사용자는 “자기 핸드 + 공개된 정보”를 계속 가져갈 수 있어야 하므로,
**Private State API** 는 반드시 있어야 한다.

## 6.8 액션 제출

### 지원 액션 타입

- `FOLD`
- `CHECK`
- `CALL`
- `BET_TO(amount)`
- `RAISE_TO(amount)`
- `ALL_IN`

### 액션 규칙

- 자신의 턴일 때만 액션 가능
- stale 상태에서 보낸 액션은 거절 가능
- `state_version` 을 같이 보내는 방식을 권장
- 중복 제출 방지를 위해 `idempotency_key` 사용 권장

### CHECK

- facing bet가 0일 때만 가능

### CALL

- 현재 최고 베팅까지 맞추는 액션
- `CALL` 과 `RAISE_TO` 는 서로 배타적

### BET_TO

- 해당 스트리트에 아직 베팅이 없을 때 사용
- 최소 `2`
- 최대 자신의 현재 스택 전체(올인)

### RAISE_TO

- 현재 최고 베팅이 있을 때 사용
- 최소값은 `ceil(current_highest_bet * 1.5)`
- 최대값은 올인 가능 총액

### ALL_IN

- 남은 스택 전부를 투입
- 최소 레이즈 미달이어도 허용

### 액션 타입 사용 분기 정리

서버는 클라이언트가 보낸 액션 타입이 현재 상황에 맞는지 검증해야 한다.

- 현재 베팅이 없는 상태에서 `RAISE_TO` → **에러** (대신 `BET_TO` 사용)
- 현재 베팅이 있는 상태에서 `BET_TO` → **에러** (대신 `RAISE_TO` 사용)
- `CALL` 제출 시 남은 스택 < `to_call` → **올인으로 자동 처리** (남은 스택 전부 투입, 사이드팟 생성)
- `BET_TO` / `RAISE_TO`의 `amount`가 남은 스택과 동일 → 정상 처리 (사실상 올인)
- `ALL_IN`은 상황 무관하게 항상 유효. 베팅이 없든 있든 남은 스택 전부 투입.

## 7. 아키텍처 요구사항

## 7.1 전체 구조

이번 버전은 **모놀리식 서버 + PostgreSQL** 구조로 충분하다.

권장 예시:

- Backend API Server: FastAPI 또는 Go 중 택1
- DB: PostgreSQL
- 배포: Railway

### 권장 구현 방향

- 서버 1개
- DB 1개
- 테이블별 상태는 메모리 캐시 가능
- 하지만 **모든 중요한 상태 변경은 반드시 DB에 기록**
- 서버 재시작 시 DB에서 active table / active hand 복구 가능해야 함

---

## 7.2 동시성 처리 원칙

가장 중요하다.

각 테이블은 액션이 순서대로 처리되어야 한다.

따라서 반드시 다음 둘 중 하나를 써야 한다.

1. **테이블별 단일 액터/큐**
2. **테이블별 mutex/lock 으로 직렬 처리**

절대로 같은 테이블의 액션을 동시에 병렬 처리하면 안 된다.

### 권장

- `table_no` 별 in-memory lock
- 액션 처리 시 DB transaction
- 처리 완료 후 snapshot 업데이트

---

## 7.3 상태 저장 원칙

핸드 도중 상태가 날아가면 안 되므로 DB가 필요하다.

### 반드시 DB에 저장해야 하는 것

- 계정
- API 키
- 칩 원장
- 테이블
- 좌석 상태
- 진행 중 핸드
- 핸드 액션 로그
- 완료된 핸드 이력
- nonce 재사용 방지 데이터

### 권장

- 액션 1개 처리될 때마다 `hand_action` 로그 저장
- 그 직후 `table_snapshot` 갱신

핸드 상태에는 최소 아래가 있어야 한다.

- hand_id
- deck 상태(외부 미노출)
- hole cards(외부 미노출)
- board
- street
- current_bet
- round_contribution
- hand_contribution
- folded / all_in
- action_seat_no
- action_deadline_at
- legal_actions 계산을 위한 값

### 9.3 팟 계산 방식

팟을 직접 mutable 구조로 복잡하게 관리하기보다,
플레이어별 총 기여금(`hand_contribution`) 기준으로 메인팟/사이드팟을 계산하는 방식을 권장한다.

이 방식이 올인/다중 올인/사이드팟 처리에서 안정적이다.

- `FORBIDDEN` — 권한 없음 (403)
- `NOT_FOUND` — 리소스 없음 (404)
- `CONFLICT` — 상태 충돌 (409, 예: 이미 착석, nonce 중복)
- `INVALID_ACTION` — 잘못된 액션 (422, 예: 자기 턴이 아님, 불가능한 액션)
- `INSUFFICIENT_BALANCE` — 잔액 부족 (422)
- `TABLE_FULL` — 좌석 없음 (422)
- `SEAT_TAKEN` — 지정 좌석 사용 중 (409)
- `STALE_STATE` — state_version 불일치 (409)
- `INTERNAL_ERROR` — 서버 오류 (500)

#### 페이지네이션

목록 조회 API(핸드 이력, 액션 로그 등)는 커서 기반 페이지네이션을 지원한다:

- 요청 파라미터: `?limit=50&cursor={last_id}`
- `limit` 기본값: 50, 최대값: 100
- 응답에 `next_cursor` 포함. `null`이면 마지막 페이지.

```json
{
  "items": [...],
  "next_cursor": "abc123",
  "has_more": true
}
```

## 11. 게임 진행 상세 규칙

## 11.1 좌석 회전

- 버튼은 다음 핸드마다 다음 활성 좌석으로 이동
- 스몰블라인드/빅블라인드는 버튼 기준으로 다음 활성 좌석에 부여
- 빈 좌석은 자동 스킵

이번 버전에서는 표준 대회용 dead button 룰보다,
**구현이 단순한 활성 좌석 기준 회전**을 우선한다.

## 11.2 프리플랍 시작

- 스몰블라인드 1 자동 포스트
- 빅블라인드 2 자동 포스트
- 홀카드 2장 배분
- 첫 액션은 빅블라인드 다음 활성 좌석부터 시작

### 헤즈업(2인) 특수 규칙

2명만 플레이할 경우 표준 헤즈업 규칙을 적용한다:

- **버튼 = 스몰블라인드** (버튼 플레이어가 SB 1칩 포스트)
- 상대방이 빅블라인드 (BB 2칩 포스트)
- **프리플랍**: 버튼(SB)이 먼저 액션
- **플랍 이후**: 버튼이 아닌 플레이어(BB)가 먼저 액션

### 블라인드 포스트 시 스택 부족

- SB/BB를 포스트할 때 스택이 블라인드 금액보다 적으면 **가진 만큼만 포스트** (올인 상태가 됨)
- 예: 스택 1인 BB → 1만 포스트, 올인 처리

## 11.3 플랍/턴/리버

- 각 스트리트에서 베팅 라운드 진행
- 종료 조건 충족 시 다음 보드 공개

## 11.4 핸드 종료 조건

- 한 명만 남으면 즉시 종료 (폴드로 인한 승리 — 카드 공개 불필요)
- 리버까지 끝나고 쇼다운 완료되면 종료
- **전원 올인 상태**: 남은 플레이어가 모두 올인이거나 1명만 액션 가능한 경우, 나머지 보드 카드를 즉시 딜하고 쇼다운 진행

## 11.5 쇼다운

- 생존 플레이어끼리 족보 비교
- 메인팟과 각 사이드팟을 독립적으로 분배
- 동점 분배 허용
- 홀수 칩(odd chip)은 **버튼에서 시계 방향(왼쪽)으로 가장 가까운 eligible seat** 에 지급

### 쇼다운 카드 공개 순서

- 이번 버전에서는 **머크(muck) 옵션 없음**. 쇼다운까지 간 모든 생존 플레이어의 카드는 자동으로 공개된다.
- 카드 공개 순서는 구현 편의상 **좌석 번호 오름차순**으로 처리해도 된다. (마지막 어그레서 우선 규칙은 이번 버전에서는 미적용)
- 공개된 카드는 핸드 이력에 기록되어 공개 API에서 조회 가능하다.

## 11.6 액션 종료 판단

다음 상황이면 베팅 라운드 종료:

- 더 이상 액션 가능한 플레이어가 없을 때
- 남은 플레이어가 모두 동일 베팅액에 도달했을 때
- 남은 플레이어가 모두 체크했을 때
- 남은 플레이어가 모두 올인이거나 폴드 상태일 때

### 액션 종료 판단 보충

- **올인 플레이어는 베팅 라운드에서 액션 대상이 아니다.** 올인한 플레이어는 이미 모든 칩을 투입했으므로 추가 액션이 불가능하다.
- 베팅 라운드 종료 후 다음 스트리트로 진입할 때, **라운드 기여금(`round_contribution`)은 0으로 리셋** 된다. `hand_contribution`은 유지.
- **1명만 액션 가능하고 나머지가 모두 올인/폴드** 인 경우: 그 1명이 콜/체크하면 (혹은 언콜 금액이 없으면) 베팅 라운드 즉시 종료.

## 12. Railway 배포 요구사항

### 기본 원칙

- Railway에 배포 가능해야 한다.
- PostgreSQL 연결을 사용한다.
- 환경변수 기반 설정이어야 한다.
- Dockerfile 기반 배포 가능하면 좋다.

### 필수 환경변수 예시

- `PORT`
- `DATABASE_URL`
- `ADMIN_PASSWORD`
- `APP_ENCRYPTION_KEY`
- `ACTION_TIMEOUT_SECONDS=600`
- `TABLE_BUYIN=40`
- `SMALL_BLIND=1`
- `BIG_BLIND=2`

### 운영 원칙

- v1은 **단일 앱 인스턴스 기준**으로 구현한다.
- 중요한 상태는 메모리가 아니라 DB에 기록한다.
- 앱 재시작 후 진행 중 테이블 복구 가능해야 한다.

### 필수 엔드포인트

- **헬스 체크**: `GET /health` — DB 연결 상태 포함. Railway 헬스 체크 설정용.
- 응답 예시: `{"status": "ok", "db": "connected"}`

### CORS 정책

- 관리자 웹 UI가 같은 도메인에서 서빙되므로 별도 CORS 설정은 불필요하다.
- 외부 봇은 API 클라이언트(HTTP 직접 호출)이므로 CORS와 무관하다.
- 필요 시 `ALLOWED_ORIGINS` 환경변수로 CORS 허용 도메인을 설정할 수 있도록 준비만 해둔다.

## 15. 수용 기준(Acceptance Criteria)

다음이 모두 만족되면 1차 완료로 본다.

### 계정/칩/키

- [ ] 관리자가 닉네임으로 계정을 생성할 수 있다.
- [ ] 계정 생성 시 기본 칩이 0이다.
- [ ] 관리자가 칩을 지급/차감할 수 있다.
- [ ] API_KEY / SECRET_KEY 발급 및 폐기가 가능하다.
- [ ] 서명 검증이 정상 동작한다.

### 테이블/좌석

- [ ] 테이블을 여러 개 생성할 수 있다.
- [ ] 각 테이블은 최대 9명까지 착석 가능하다.
- [ ] 플레이어는 특정 테이블 번호를 지정해 착석할 수 있다.
- [ ] 플레이어는 특정 테이블에서 일어설 수 있다.
- [ ] 잔액이 40 미만이면 착석 실패한다.

### 게임 진행

- [ ] 블라인드 1/2로 핸드가 시작된다.
- [ ] 착석 시 테이블 스택 40으로 시작한다.
- [ ] 체크, 콜, 베팅, 레이즈, 올인, 폴드가 정상 동작한다.
- [ ] 체크레이즈가 가능하다.
- [ ] CALL 후 같은 턴에 이어서 RAISE 하는 복합 액션은 불가능하다.
- [ ] 최소 레이즈 규칙이 `ceil(current_highest_bet * 1.5)` 로 동작한다.
- [ ] 올인과 사이드팟이 정확히 동작한다.
- [ ] 10분 무응답 시 자동 폴드가 실행된다.

### 조회/API

- [ ] 플레이어는 자기 홀카드 + 공개 정보를 조회할 수 있다.
- [ ] 공개 API에서는 타인 비공개 카드가 노출되지 않는다.
- [ ] 테이블별 상태를 조회할 수 있다.
- [ ] 테이블별 완료 핸드 이력을 조회할 수 있다.
- [ ] 테이블별 액션 이력을 조회할 수 있다.

### 저장/복구

- [ ] 모든 액션이 DB에 저장된다.
- [ ] 칩 변동이 원장에 기록된다.
- [ ] 서버 재시작 후 진행 중 상태 복구가 가능하다.

## 19. 이번 문서 기준으로 확정된 해석

모호한 부분은 아래처럼 확정한다.

1. **계정 생성 시 칩 자동 지급 없음**
2. **관리자가 칩 지급**
3. **착석 시 40칩 고정 바이인**
4. **외부 봇은 API_KEY/SECRET_KEY 서명으로 인증**
5. **한 계정은 동시에 한 테이블만 착석**
6. **10분 무응답이면 무조건 자동 폴드**
7. **체크레이즈 허용 / 콜레이즈 금지**
8. **최소 레이즈 총액 = ceil(현재 최고 베팅 × 1.5)**
9. **사이드팟 필수**
10. **게임 이력과 액션 이력은 테이블별로 조회 가능해야 함**
11. **DB는 PostgreSQL 사용**
12. **배포 대상은 Railway**
13. **v1은 정식 MTT 엔진이 아니라 멀티테이블 실시간 홀덤 플랫폼**
14. **카드 표기: 2글자 (랭크+수트), 예: `As`, `Td`, `2c`**
15. **좌석 번호 범위: 1~9**
16. **헤즈업(2인) 시 버튼 = SB, 프리플랍은 버튼 먼저 액션**
17. **핸드 진행 중 착석한 플레이어는 다음 핸드부터 참여**
18. **타임스탬프는 Unix epoch 초 단위 (정수 10자리)**
19. **nonce TTL: 10분 이상 경과 시 삭제 가능**
20. **쇼다운 시 머크 불가, 모든 생존 플레이어 카드 자동 공개**
21. **모든 칩 연산은 정수(integer), 소수점 없음**
22. **전원 올인 시 나머지 보드 카드 즉시 딜 후 쇼다운**
23. **CLOSED 테이블은 재오픈 불가**
24. **is_sitting_out 기능은 이번 버전에서 미구현**
25. **지갑 잔액은 음수 불가**
26. **레이즈 횟수 제한 없음 (No-Limit)**
27. **관리자 API 인증은 세션 쿠키 또는 Bearer 토큰 (ADMIN_PASSWORD) 방식**
